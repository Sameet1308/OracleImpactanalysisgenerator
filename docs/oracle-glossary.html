T

TABLE Foundation

Stores persistent data — the foundation of every Oracle ERP system

What is it?

A TABLE is a structured collection of rows and columns that stores business data. In Oracle ERP, tables hold master data (employees, departments, jobs), transactional data (payroll, invoices), and audit logs. Tables are the most critical artifacts because everything else depends on them.

Who depends on a TABLE?

  • VIEW reads from it (SELECT ... FROM table)
  • PROCEDURE writes to it (INSERT/UPDATE/DELETE)
  • FUNCTION reads from it (SELECT ... INTO ...)
  • TRIGGER fires on it (BEFORE/AFTER INSERT/UPDATE/DELETE)
  • SEQUENCE generates IDs for it (EMP_ID_SEQ.NEXTVAL)
  • OIC FLOW reads/writes via SQL or procedure calls
  • BIP REPORT queries it in dataset SQL
  • GROOVY accesses it via executeQuery/executeProcedure

Risk when modified

Criticality: 10/10 — Changing a table's columns, constraints, or structure can break every artifact that references it. A single column rename can cause ORA-00904 (invalid identifier) in views, ORA-06508 in packages, and silent data loss in OIC flows.

Sample Code: EMPLOYEES table

CREATE TABLE EMPLOYEES ( employee_id NUMBER(10) PRIMARY KEY, first_name VARCHAR2(50) NOT NULL, last_name VARCHAR2(50) NOT NULL, email VARCHAR2(100) UNIQUE, hire_date DATE DEFAULT SYSDATE, salary NUMBER(12,2), commission_pct NUMBER(4,2), department_id NUMBER(10) REFERENCES DEPARTMENTS(department_id), job_id NUMBER(10) REFERENCES JOBS(job_id) );
V

VIEW Read Layer

Virtual table defined by a SELECT query — reads from one or more tables

What is it?

A VIEW is a stored SQL query that acts like a virtual table. It doesn't store data itself — it reads from underlying tables every time it's queried. Views simplify complex joins, enforce security (expose only certain columns), and provide a stable interface even when underlying tables change.

Dependencies

  • Depends on: TABLE — the source tables in FROM/JOIN clauses
  • Depended on by: OIC FLOW, BIP REPORT, PROCEDURE — anything that queries the view

Risk when source table changes

Criticality: 7/10 — If a source table column is renamed or dropped, the view becomes invalid (ORA-04063: view has errors). All downstream consumers (OIC, BIP, procedures) that query this view will also fail.

Sample Code: HR_EMPLOYEE_SUMMARY view

CREATE OR REPLACE VIEW HR_EMPLOYEE_SUMMARY AS SELECT e.employee_id, e.first_name || ' ' || e.last_name AS full_name, e.email, e.hire_date, e.salary, d.department_name, j.job_title FROM EMPLOYEES e JOIN DEPARTMENTS d ON e.department_id = d.department_id JOIN JOBS j ON e.job_id = j.job_id WHERE e.hire_date >= ADD_MONTHS(SYSDATE, -24);
P

PROCEDURE Action

Named PL/SQL block that performs actions — INSERT, UPDATE, DELETE on tables

What is it?

A PROCEDURE is a reusable PL/SQL program stored in the database. Unlike functions, procedures don't return a value — they perform actions like inserting records, updating salaries, or calling other procedures. They're the "workhorses" of Oracle ERP business logic.

Dependencies

  • Depends on: TABLE (reads/writes), FUNCTION (calls), VIEW (queries)
  • Depended on by: OIC FLOW (invoked as step), GROOVY (executeProcedure), PACKAGE (grouped inside)

Sample Code: GET_EMPLOYEE_SALARY

CREATE OR REPLACE PROCEDURE GET_EMPLOYEE_SALARY ( p_employee_id IN NUMBER, p_salary OUT NUMBER, p_bonus OUT NUMBER ) AS BEGIN SELECT salary, salary * NVL(commission_pct, 0) INTO p_salary, p_bonus FROM EMPLOYEES WHERE employee_id = p_employee_id; EXCEPTION WHEN NO_DATA_FOUND THEN p_salary := 0; p_bonus := 0; END GET_EMPLOYEE_SALARY;
f

FUNCTION Compute

PL/SQL block that computes and RETURNS a value — used in SQL and other PL/SQL

What is it?

A FUNCTION is like a procedure but it RETURNS a value. Functions can be called inside SQL queries (SELECT my_func() FROM dual), making them essential for computed columns, validation logic, and business rules. In Oracle ERP, functions calculate bonuses, tax rates, eligibility, etc.

Dependencies

  • Depends on: TABLE (reads data for computation)
  • Depended on by: GROOVY (executeFunction), VIEW (in SELECT), PROCEDURE (called from)

Sample Code: CALC_ANNUAL_BONUS

CREATE OR REPLACE FUNCTION CALC_ANNUAL_BONUS ( p_employee_id IN NUMBER ) RETURN NUMBER AS v_salary NUMBER; v_commission NUMBER; v_bonus NUMBER; BEGIN SELECT salary, NVL(commission_pct, 0) INTO v_salary, v_commission FROM EMPLOYEES WHERE employee_id = p_employee_id; v_bonus := v_salary * (0.1 + v_commission); RETURN ROUND(v_bonus, 2); END CALC_ANNUAL_BONUS;
Pk

PACKAGE Container

Groups related procedures and functions — the Oracle equivalent of a class/module

What is it?

A PACKAGE is two objects: a SPEC (public interface declaring procedure/function signatures) and a BODY (private implementation). Packages group related logic — e.g., HR_PACKAGE contains GET_HEADCOUNT and SYNC_EMPLOYEE. They're critical because many external systems reference PACKAGE.FUNCTION notation.

Dependencies

  • Depends on: TABLE, VIEW, FUNCTION, PROCEDURE
  • Depended on by: OIC FLOW, GROOVY, BIP — all call PKG.FUNC()

Sample Code: HR_PACKAGE

-- Package Specification (public interface) CREATE OR REPLACE PACKAGE HR_PACKAGE AS FUNCTION GET_HEADCOUNT(p_dept_id NUMBER) RETURN NUMBER; PROCEDURE SYNC_EMPLOYEE(p_emp_id NUMBER, p_data VARCHAR2); END HR_PACKAGE; / -- Package Body (implementation) CREATE OR REPLACE PACKAGE BODY HR_PACKAGE AS FUNCTION GET_HEADCOUNT(p_dept_id NUMBER) RETURN NUMBER AS v_count NUMBER; BEGIN SELECT COUNT(*) INTO v_count FROM EMPLOYEES WHERE department_id = p_dept_id; RETURN v_count; END; PROCEDURE SYNC_EMPLOYEE(p_emp_id NUMBER, p_data VARCHAR2) AS BEGIN UPDATE EMPLOYEES SET last_name = p_data WHERE employee_id = p_emp_id; COMMIT; END; END HR_PACKAGE;
Tr

TRIGGER Event Hook

Automatically fires before/after INSERT, UPDATE, DELETE on a table

What is it?

A TRIGGER is PL/SQL code that runs automatically when a DML event occurs on a table. Used for audit logging, data validation, cascading updates. Triggers are "invisible" — they run without being explicitly called, making them dangerous if broken.

Sample Code: PAYROLL_TRIGGER

CREATE OR REPLACE TRIGGER PAYROLL_TRIGGER AFTER UPDATE OF salary ON EMPLOYEES FOR EACH ROW BEGIN INSERT INTO SALARY_AUDIT_LOG ( employee_id, old_salary, new_salary, changed_date, changed_by ) VALUES ( :NEW.employee_id, :OLD.salary, :NEW.salary, SYSDATE, USER ); END PAYROLL_TRIGGER;
#

SEQUENCE ID Generator

Auto-incrementing number generator for primary keys

What is it?

A SEQUENCE generates unique sequential numbers, typically used for primary key columns. In Oracle ERP, sequences like EMP_ID_SEQ generate employee IDs. They're low-risk individually but essential for INSERT operations.

Sample Code

CREATE SEQUENCE EMP_ID_SEQ START WITH 1000 INCREMENT BY 1 NOCACHE NOCYCLE; -- Usage in INSERT: INSERT INTO EMPLOYEES (employee_id, ...) VALUES (EMP_ID_SEQ.NEXTVAL, ...);
OIC

OIC Integration Flow External

Oracle Integration Cloud — orchestrates data movement between cloud and on-prem systems

What is it?

An OIC flow is an integration defined in Oracle Integration Cloud. It connects Oracle ERP to other systems — syncing employees from HCM Cloud to a database, triggering workflows, transforming data. OIC flows call database procedures, query views, and write to tables via connections.

Critical: OIC flows are completely INVISIBLE to Oracle DBA_DEPENDENCIES. A change to EMPLOYEES table can break an OIC flow and nobody in the database team would know.

Sample Code: HR_EMPLOYEE_SYNC

<integration name="HR_EMPLOYEE_SYNC" type="scheduled_orchestration"> <schedule>0 2 * * *</schedule> <!-- Daily at 2 AM --> <connection name="HCM_ADAPTER" type="oracle_hcm_cloud" /> <connection name="HR_DB_CONN" type="oracle_db" /> <steps> <step name="FETCH_EMPLOYEES"> <query>SELECT * FROM HR_EMPLOYEE_SUMMARY WHERE hire_date >= SYSDATE - 30</query> </step> <step name="CALC_SALARY"> <procedure>GET_EMPLOYEE_SALARY</procedure> </step> <step name="UPSERT_EMPLOYEES"> <target>EMPLOYEES</target> </step> </steps> </integration>
BIP

BI Publisher Report Reporting

Oracle BI Publisher — generates PDF, Excel, HTML reports from database queries

What is it?

A BIP report defines data models (SQL queries) and layout templates that produce formatted reports. The data model contains embedded SQL that queries tables and views directly. If a source table changes, the report's SQL fails silently — no compilation error, just wrong or missing data.

Sample Code: MONTHLY_PAYROLL_REPORT

<report name="MONTHLY_PAYROLL_REPORT"> <dataModel> <dataset name="PayrollByDept"> <sql> SELECT d.department_name, COUNT(e.employee_id) AS headcount, SUM(e.salary) AS total_salary, ROUND(AVG(e.salary), 2) AS avg_salary FROM PAYROLL_VIEW e JOIN DEPARTMENTS d ON e.department_id = d.department_id GROUP BY d.department_name ORDER BY total_salary DESC </sql> </dataset> </dataModel> <output formats="pdf,xlsx,html" /> </report>
G

Groovy Script HCM/ERP Cloud

Oracle Fusion HCM/ERP Cloud — Groovy-based business rules, fast formulas, personalizations

What is it?

Groovy scripts run inside Oracle Fusion Cloud (HCM, ERP, SCM) for custom business logic. They call database functions/procedures via executeFunction() and executeQuery(). They're the hardest to track because they live in the cloud application, not the database.

Sample Code: COMPENSATION script

// Oracle Fusion HCM — Compensation Calculation Rule // Runs during annual compensation cycle def employeeId = getCurrentEmployeeId() def departmentId = getCurrentDepartmentId() // Call PL/SQL function to calculate annual bonus def annualBonus = sql.executeFunction( "CALC_ANNUAL_BONUS", [employeeId] ) // Get department headcount via package function def headcount = sql.executeFunction( "HR_PACKAGE.GET_HEADCOUNT", [departmentId] ) // Calculate per-capita department adjustment def deptBudget = 500000.00 def perCapitaAdj = (headcount > 0) ? deptBudget / headcount : 0 def adjustedBonus = annualBonus + (perCapitaAdj * 0.05) return [ employee_id: employeeId, annual_bonus: annualBonus, dept_headcount: headcount, adjustment: Math.round(adjustedBonus * 100) / 100 ]
