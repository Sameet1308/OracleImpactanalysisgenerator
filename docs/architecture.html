Client Tier

Authentication: Oracle IDCS SSO integration
Frontend: Next.js 15, React 19, D3.js radial spoke graph, Ask Ora1 conversational AI chat with file attach

Security: Session-based auth. HTTPS enforced. No credentials stored client-side.
Rendering: Server-side rendering (SSR) for initial load. Client-side D3.js for graph interactions.
Chat: Maintains conversation history client-side. Sends full context with each request for stateless backend.

API Gateway + Validation

8 REST endpoints with CORS, rate limiting
Input validation: file type whitelist (.sql/.xml/.groovy), size limits, schema check before parsing

Endpoints: /upload /demo /analyze /chat /graph /objects /report /token
Guardrail: Rejects non-Oracle file types. Validates XML well-formedness before parsing. Caps upload size.

Ingestion Service

Auto-detect parser by file extension
Extensible: add new Oracle product parsers (EBS, JDE, EPM) without changing dispatch logic

SQL: sqlparse tokenization + custom regex for CREATE, FK (multi-line), FROM/JOIN, function calls
XML: xml.etree for OIC integration flows, BIP data models, embedded SQL extraction
Groovy: regex for executeFunction, executeProcedure, executeQuery
Extensibility: PARSER_MAP dispatch — register new parser function per file extension. Zero changes to existing code.

Graph Service

NetworkX DiGraph — BFS blast radius traversal
Risk scoring: 52% direct + 18% indirect + 30% type criticality
Impact cache avoids recomputation

Type weights: TABLE:10, PKG:9, PROC:8, OIC:8, VIEW:7, BIP:7, GROOVY:6
Severity: CRITICAL ≥80 | HIGH ≥60 | MEDIUM ≥40 | LOW <40
Verified: EMPLOYEES → score 89 CRITICAL with 8 direct + 5 indirect dependents

RAG Service

Local embeddings — all-MiniLM-L6-v2 (384-dim, zero API cost)
ChromaDB vector store with 2-stage retrieval

Chunking: SQL by CREATE block (2000 chars). XML by steps/datasets. Groovy by function blocks.
Stage 1: Exact metadata match (object_name filter) — score 1.0
Stage 2: Semantic similarity search on impacted object names — cosine score
Token optimization: Only top-5 chunks (500 chars each) included in prompt. Avoids token waste.

AI Service + Guardrails

Prompt Builder: merges impact + code + question into single optimized prompt
Token Manager: JWT auto-refresh, expiry detection, usage tracking
Guardrails: prompt size cap, PII filtering, output structure validation

Prompt structure: System context (50 tokens) + Impact data (~200 tokens) + Code snippets (5×500 chars) + Question
Token budget: Total prompt capped at ~3000 tokens. Conversation history limited to last 6 turns.
Output validation: Response parsed into 4 mandatory sections. If any section missing, re-parsed with fallback extraction.
PII filter: Strips credentials, connection strings, passwords from code context before sending to AI.
Cost tracking: Each API call logged with token count, latency, success/failure.

Output Service

4 AI deliverables: Root Cause, Fix Recommendations (ORA codes), Testing Checklist, Rollback Plan
PDF export via ReportLab

Root Cause: Explains WHY the change breaks downstream artifacts with specific object references
Fix Recs: Includes ORA error codes (ORA-04063, ORA-06508, ORA-00904) with remediation commands
Testing: Generates regression test plan for each impacted object
Rollback: Step-by-step reversal with DBMS_METADATA, ALTER COMPILE, version control strategy

Data Tier (Production)

Oracle 23ai: Graph store + vector search in one database
OCI Object Storage: Persistent artifact storage

Graph persistence: Oracle 23ai property graph for dependency relationships
Vector persistence: Oracle 23ai vector search replaces ChromaDB for production-scale RAG
Artifact storage: OCI Object Storage for uploaded files with versioning
Why Oracle 23ai: Single database for graph + vector + relational = no separate ChromaDB/Neo4j needed

Observability

Structured logging of all API calls and parse results
Token cost tracking per query with latency monitoring
Health check API for AI mode status

Logs: Every upload, parse, analyze, chat call logged with duration, object count, error details
Token metrics: BlueVerse calls tracked with estimated token count, response time, cost per interaction
Health: GET / returns AI mode (live/expired), objects loaded, token status — used by frontend health indicator
