INGESTION LAYER
SQL Parser
parsers/sql_parser.py — sqlparse + regex
CREATE TABLE/VIEW/PROC/FUNC/PKG/TRIGGER/SEQ. FK detection (multi-line). FROM/JOIN refs. Function calls.
Input:
raw .sql file content
Output:
objects[] + dependencies[]
Patterns:
CREATE_PATTERN, FK_PATTERN (position-based for multi-line), TABLE_REF_PATTERN, CALL_PATTERN, SEQ_PATTERN, TRIGGER_ON_PATTERN
Tested on:
Oracle official HR schema (559 lines) → 11 objects, 12 FK deps, 100% accuracy on standard DDL.
OIC/BIP Parser
parsers/oic_parser.py — xml.etree
Auto-detects <integration> vs <report>. Extracts steps, connections, embedded SQL, datasets.
OIC flow:
flow name, connections (OIC_CONNECTION), procedure calls, query table refs, target writes
BIP report:
report name, dataset SQL parsing, function calls
SQL keyword filtering:
excludes SELECT, WHERE, COUNT, SUM etc. to avoid false positives
Groovy Parser
parsers/groovy_parser.py — regex
executeQuery, executeFunction, executeProcedure. Embedded SQL table refs.
Patterns:
EXEC_QUERY, EXEC_FUNC, EXEC_PROC, SQL_TABLE in string literals
Key insight:
Groovy never touches tables directly — only calls PL/SQL. Creates indirect dependencies invisible to DBA_DEPENDENCIES.
objects[] + dependencies[] + source code
ANALYSIS LAYER (parallel processing)
Dependency Graph
graph/engine.py — NetworkX DiGraph
Directed graph. Nodes=objects, Edges=deps. BFS blast radius. Risk score 0-100.
Score formula:
52% × min(direct×13, 100) + 18% × min(indirect×22, 100) + 30% × type_weight×10
Type weights:
TABLE:10, PKG:9, PROC:8, OIC:8, VIEW:7, BIP:7, FUNC:8, GROOVY:6
Severity:
CRITICAL ≥80 | HIGH ≥60 | MEDIUM ≥40 | LOW <40
EMPLOYEES:
8 direct, 5 indirect, TABLE=10 → score 89 CRITICAL
↓ impact_data {score, severity, direct[], indirect[]}
RAG Knowledge Base
knowledge/rag.py — sentence-transformers + ChromaDB
Embeds source code (384-dim, local). 2-stage retrieval: exact match + semantic search. Top-k=5.
Embedding model:
all-MiniLM-L6-v2 — runs locally, no API calls, zero cost
Vector store:
ChromaDB (in-memory, ephemeral). Cosine similarity.
Chunking:
SQL: by CREATE block (2000 chars). XML: full file + steps. Groovy: full file + functions.
Retrieval:
Stage 1: metadata filter (object_name == target) → score 1.0
Stage 2: semantic search on "{target} {impacted_objects}" → cosine score
Merge, deduplicate, return top 5 by relevance.
↓ code_context [{text, source_file, score}]
MERGE: impact_data + code_context + user_question → single prompt
AI LAYER
Prompt Builder
ai/oci_genai.py — _build_prompt()
Constructs one prompt string from: system context + impact stats + code snippets (5 × 500 chars) + user question. No chains, no agents.
Prompt structure:
1. System: "You are Oracle Pythia-26, an Oracle ERP impact analysis assistant..."
2. Impact: "TARGET: EMPLOYEES (TABLE), SCORE: 89, SEVERITY: CRITICAL, DIRECT: [8 names], INDIRECT: [5 names]"
3. Code: "## RELEVANT SOURCE CODE\n[EMPLOYEES (TABLE)]: CREATE TABLE EMPLOYEES (employee_id NUMBER..."
4. Question: "Provide analysis in 4 sections: ROOT CAUSE, FIX RECOMMENDATIONS, TESTING CHECKLIST, ROLLBACK PLAN"
Response parsing:
Splits by section headers → extracts numbered lists → structured JSON output.
→ BlueVerse Foundry
ai/blueverse.py — call_blueverse()
One HTTP POST. 30s timeout. Returns text response. Parsed into 4 sections.
Endpoint:
POST blueverse-foundry.ltimindtree.com/chatservice/chat
Payload:
{space_name: "AI_Elite_ora1_45e8873c", flowId: "69ba...", question: full_prompt}
Auth:
JWT Bearer token. Expires ~20min. Runtime refresh via POST /api/token.
Fallback:
If token expired → returns None immediately (no 30s wait) → mock engine generates deterministic response with real ORA error codes.
OUTPUT
Root Cause Analysis
Why this change breaks downstream artifacts
Fix Recommendations
ORA error codes + remediation steps
Testing Checklist
Regression test plan for impacted objects
Rollback Plan
Safe reversal steps if deployment fails
EXTERNAL — BlueVerse Foundry (LTIMindtree Cloud)
AI_Elite_Ora1 Agent
Pre-trained Oracle ERP agent on BlueVerse Marketplace
Receives single prompt. Returns Oracle-specific analysis with ORA error codes. Stateless — no conversation memory on server side.
Why BlueVerse over direct LLM:
Pre-trained on Oracle ERP domain. Knows ORA error codes, DBMS_METADATA patterns, ALTER COMPILE syntax, Oracle best practices.
Limitation:
Stateless API — no conversation history. We reconstruct full context in every prompt (client-side history management).
Mock Fallback
ai/oci_genai.py — _mock_analysis()
Deterministic rule-based output when BlueVerse unavailable. References real object names + ORA codes. Demo always works.

Key Differentiator

Oracle DBA_DEPENDENCIES only tracks database objects.

AI_Elite_Ora1 crosses the boundary:
OIC flows + BIP reports + Groovy scripts
= dependencies invisible to Oracle's own tools.

Known Limitations

Production: add DBMS_METADATA as complementary source
In-memory only, no persistence
