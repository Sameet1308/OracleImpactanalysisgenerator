Risk score: ninety-seven out of a hundred. Severity: CRITICAL. Nine direct impacts. Four indirect. Every red path on the graph is a dependency that would break if I change this table.

The scoring formula weights direct dependencies at fifty-two percent, indirect at eighteen percent, and object-type criticality at thirty percent. So a heavily-referenced table like EMPLOYEES lands in CRITICAL. A standalone sequence would be LOW. The number is deterministic — same input, same score, every time. Reproducible. Auditable.

BlueVerse AI Layer

65 sec · AI governance + grounding
close column modal → click CRITICAL risk pill → AI modal opens
Now the AI. We route through BlueVerse Foundry — LTM's sanctioned AI platform. The agent we use is called AI_Elite_Ora1, Oracle-context tuned.

What's grounding the response is a RAG pipeline. We run ChromaDB locally inside the container using sentence-transformers — source code never leaves the process. Before the prompt goes to BlueVerse, a PII filter strips credentials, connection strings, JWTs, and emails via regex. We cap prompts at twelve kilobytes to prevent token bloat.

The agent returns four sections. Root cause — a narrative. Five fix steps, each citing real ORA error codes. Testing checklist. Five-step rollback plan. We run hallucination detection on every response, cross-checking ORA codes against a whitelist.

If BlueVerse goes down, we fall through to a deterministic local engine — zero calls to Claude, OpenAI, Gemini, or OCI. LTM BlueVerse-only policy, by design.
